# Supply-chain verification

How to verify that the `orionfold-relay` you install is exactly what this
repository's CI built — before you run it. Everything on this page is
checkable from your side with standard npm and GitHub tooling; none of it
requires trusting a claim we make.

## What ships, and from where

| Artifact | Channel | Integrity mechanism |
|---|---|---|
| npm package `orionfold-relay` (CLI + app source) | registry.npmjs.org | npm provenance attestation (SLSA v1), registry signatures |
| Production server build (`relay-next-build-<version>.tgz`) | GitHub Release asset on [orionfold/relay](https://github.com/orionfold/relay/releases) | SHA-256 sidecar, verified by the CLI before extraction |
| CycloneDX SBOM (`orionfold-relay-<version>.sbom.cdx.json`) | GitHub Release asset | Attached by the same gated publish workflow |

## Provenance: the package is built from this repo, by CI, with no tokens

Publishing happens **only** via [npm Trusted Publishing (OIDC)](https://docs.npmjs.com/trusted-publishers)
from the committed workflow [`.github/workflows/publish.yml`](../../.github/workflows/publish.yml),
triggered by a version tag. There is no npm token anywhere — not in CI
secrets, not on a maintainer laptop — so there is no token to steal. npm
generates a [SLSA v1 provenance attestation](https://slsa.dev/spec/v1.0/provenance)
for every publish, binding the tarball to the exact repo, commit, and
workflow run that produced it.

Verify it yourself:

```bash
# In a project that has orionfold-relay installed:
npm audit signatures
# → "NNN packages have verified registry signatures / attestations"

# Or inspect the attestation directly, no install needed:
npm view orionfold-relay dist.attestations
```

The npm package page also renders the green "Built and signed on GitHub
Actions" provenance badge with a link to the workflow run.

## SBOM: audit the dependency tree without installing it

Every release from **0.20.0** onward attaches a
[CycloneDX](https://cyclonedx.org/) SBOM of the production dependency tree
(dev dependencies omitted) as a GitHub Release asset:

```
https://github.com/orionfold/relay/releases/download/v<version>/orionfold-relay-<version>.sbom.cdx.json
```

It is generated by `npm sbom` inside the publish workflow, from the same
committed lockfile the published package was built and tested against — so
what the SBOM says is what `npm install orionfold-relay@<version>` resolves.
Feed it to your scanner of choice (Dependency-Track, Grype, osv-scanner).

## The production build artifact is checksum-verified before use

The npm package is deliberately small (a size guard in the publish workflow
fails the release if it exceeds 10 MB). On first launch of a version, the
CLI downloads the prebuilt Next.js server build for **that exact version**
from the GitHub Release and verifies it against its `.sha256` sidecar before
extracting ([`src/lib/desktop/prebuilt-download.ts`](../../src/lib/desktop/prebuilt-download.ts));
a mismatch aborts loudly. Verified downloads are cached per version. For
air-gapped installs, `RELAY_BUILD_ARTIFACT_URL` accepts a `file://` or
mirror URL to an artifact you fetched out-of-band — the `.sha256` sidecar is
still required next to it and still verified.

## Pinning a version

Nothing in Relay self-updates. To stay on an evaluated version:

```bash
# One-off run, exact version:
npx orionfold-relay@0.19.0

# Global install, exact version:
npm install -g orionfold-relay@0.19.0
```

Two properties make the pin meaningful:

- **The version you pin is the version that runs.** The production build
  artifact is fetched by exact version and checksum-verified; there is no
  "latest" channel, no auto-upgrade, no update check.
- **The stack under it is pinned too.** `next` is pinned to an exact version
  in `package.json` (not a `^` range), and releases are built and smoke-tested
  from the committed `package-lock.json`.

Upgrades are always an explicit act on your side: install the new version,
re-verify provenance and SBOM if your process requires it.

## Release gating (what must pass before anything publishes)

The publish workflow refuses to release unless, in order: the tag matches
`package.json`; the CLI builds; the licensing test suite passes; the tarball
size guard passes; the production build compiles; and a full customer
simulation succeeds — pack the tarball, install it into a clean directory,
run the first-launch artifact download, serve with `next start`, and exercise
the free→licensed journey against a real signed license. A failure at any
step means no publish. See [`docs/RELEASING.md`](../RELEASING.md).
